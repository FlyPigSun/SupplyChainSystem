# 匹配修正

> 一句话：记住用户人工修正的原料-价格匹配关系，后续用量计算和成本核查时自动应用，减少重复修正。

## 核心功能

- 保存人工修正记录（source → target_price_id）
- 查询所有修正记录
- 批量查找修正（按 source 列表批量查询）
- 删除修正记录
- 自动应用于用量计算和成本核查

## 核心逻辑

### 修正记录结构
```sql
CREATE TABLE match_corrections (
  id INTEGER PRIMARY KEY,
  source_name TEXT NOT NULL,          -- 原料名称
  source_brand_spec TEXT DEFAULT '',  -- 品牌规格
  source_supplier TEXT DEFAULT '',    -- 供应商
  target_price_id INTEGER,            -- 指向 material_prices.id
  created_by TEXT,
  created_at DATETIME,
  UNIQUE(source_name, source_brand_spec, source_supplier)
)
```

### 保存修正
```sql
INSERT OR REPLACE INTO match_corrections (source_name, source_brand_spec, source_supplier, target_price_id, created_by, created_at)
VALUES (?, ?, ?, ?, ?, datetime('now'))
```
- `INSERT OR REPLACE`：同一 source 重复修正时覆盖旧记录
- 校验 `target_price_id` 必须存在于 `material_prices`

### 自动应用（在 calculator.js 和 bomCheck.js 中）
1. 一次性加载全部修正记录
2. 构建 `correctionMap`，key 为 `{source_name}||{source_brand_spec}||{source_supplier}`
3. 每个原料匹配后，查询 correctionMap
4. 如命中：替换 `priceInfo` 为修正目标，`matchType` 变为 `corrected`

### 匹配引擎（priceMatcher.js）

#### 用量计算匹配（matchPrice）
4 级优先级：

| 优先级 | matchType | 规则 |
|--------|-----------|------|
| 1 | `exact` | brand + supplier 完全一致 |
| 2 | `brand_model` | brand + model 匹配（精确或相似） |
| 3 | `supplier_model` | supplier + unified_name/model 匹配 |
| 4 | `fuzzy` | 仅型号匹配，品牌不同 |

辅助函数：
- `extractBrandAndModel()`：从原料名称中拆分品牌和型号
- `isSimilar()`：字符串相似度（包含关系 + 关键词覆盖率 ≥ 60%）
- `normalize()`：去掉括号、空格、后缀码
- `detectFlavorDiff()`：检测口味关键词差异

#### 成本核查匹配（matchPriceForAudit）
与用量计算场景不同，输入是 Excel 中的 `(原料名称, 品牌规格)` 组合，额外支持：
- `flavor_diff`：品牌匹配但口味不同
- 更细粒度的匹配提示和 warnings

### 通用词过滤
以下单字不做模糊匹配：`水、盐、油、糖、粉、奶、酱、蛋、面、肉、酒、醋、茶`

### 字母数字编码保护
如 `YF042`、`MS01T` 等编码只做精确匹配，不做模糊匹配。

## 数据流

```
用户在 CorrectionDialog 选择目标价格
              │
              ▼
    POST /api/match-corrections
              │
              ▼
        match_corrections (INSERT OR REPLACE)
              │
              ▼
    ┌─────────────────┐
    │  calculator.js   │──► 用量计算时自动查询并应用
    │  bomCheck.js     │──► 成本核查时自动查询并应用
    └─────────────────┘
```

## 关键设计决策

- 修正记录以 `(source_name, source_brand_spec, source_supplier)` 为唯一键，覆盖同一原料不同规格/供应商的场景
- `INSERT OR REPLACE` 保证重复修正时自动覆盖，无需先删除
- 批量加载修正记录避免 N+1 查询问题
- 匹配引擎与修正系统解耦：匹配引擎只做匹配，修正系统在匹配后叠加
- 字母+数字编码（如 YF042）视为品牌专属，精确匹配避免误配
